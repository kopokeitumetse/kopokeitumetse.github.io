How I Validate Client Data Before a Sage 50c Go-Live

2024-11-18 | 8 min read | tutorial sage-erp data-migration

After implementing Sage 50c Pastel for 20+ clients at Businesoft, the single most common cause of a rocky go-live was not configuration errors or user training gaps — it was bad data getting into the system. A chart of accounts with duplicate codes. An opening balance that doesn't reconcile with the audited trial balance. A supplier record that was imported twice with slightly different names.

These are fixable problems, but fixing them after go-live is expensive, embarrassing, and erodes client trust immediately. Fixing them before go-live costs a few hours of careful work. This post walks through the validation checklist I developed and used on every migration.

1. Understand What You're Migrating

Before touching any data, I sit down with the client's accountant or finance manager and agree on the scope. Sage 50c holds a lot — you don't necessarily migrate all of it on day one.

Typical migration scope for a new Sage 50c implementation:
  • Chart of Accounts (CoA) — categories, account codes, VAT links
  • Supplier and Customer master data — names, tax numbers, payment terms, contact details
  • Inventory items — descriptions, units of measure, selling prices, tax categories
  • Opening balances — as at the agreed cut-over date (usually end of a financial period)

Historical transactions (invoices, payments, journals from prior years) are almost never migrated — they stay in the old system for reference. Trying to migrate transaction history into Sage is where migrations go badly wrong and usually isn't worth the time. Agree this boundary upfront.

2. Extract and Audit the Source Data

Most clients hand you data in one of three states: a well-maintained Excel workbook, a partially-maintained Excel workbook, or a CSV export from an old accounting package like Pastel Evolution, QuickBooks, or — the most interesting one — a set of manual ledger books that someone has started digitising halfway through.

The first thing I do is load the raw source file into Excel and run a quick audit pass: 

-- SQL-style logic, but done in Excel pivot tables / filters

Check 1: Duplicates in key columns
Supplier Name   → count where name appears > 1 time
Account Code    → count where code appears > 1 time
Tax Reg Number  → count where TRN appears > 1 time (flag nulls too)

Check 2: Missing required fields
COUNTBLANK(AccountCode column) → must be zero before import
COUNTBLANK(SupplierName column) → must be zero

Check 3: Data type consistency
AccountCode → should be numeric or alphanumeric in a consistent format
OpeningBalance → should be numeric, not "1,234.00 BWP" or "N/A"
PaymentTerms → should map to Sage valid values: 0, 7, 14, 30, 60

This pass usually surfaces 10–30 issues per client. I log each one in a shared Google Sheet (or Word doc if the client doesn't have Google Workspace) and work through them with the accountant before proceeding.

3. Map Source Fields to Sage Fields

Sage 50c's import templates have specific column requirements. The field names and accepted formats are not always intuitive. Here's a mapping example for the Supplier master import: 

Source column          → Sage Import Field         Notes
Supplier Name          → Supplier_Name             Max 40 chars — truncate if longer
Tax Reg Number         → Tax_Reg_No                Format: BW + 9 digits (Botswana)
Account Code           → Acc_Code                  Unique, max 8 chars
Contact Person         → Contact_1                 Optional but useful
Phone                  → Tel                       Digits only, no spaces
Payment Terms (days)   → Terms                     Integer: 0, 7, 14, 30, 60
Opening Balance        → Opening_Balance           Positive = credit (you owe them)
Opening Balance Date   → Opening_Balance_Date      DD/MM/YYYY
Watch out: Opening balances in Sage are entered from the perspective of what the business owes or is owed. A supplier opening balance of R5,000 means the business owes the supplier R5,000. Clients sometimes present this inverted. Always confirm the sign convention before importing.

4. Prepare the Import File

Once the mapping is agreed, I build the import CSV from the cleaned source data. I do this in Excel using a staging sheet — a separate tab that pulls from the cleaned data and transforms it into the exact Sage import format. This way the original data stays untouched and auditable. 

Staging sheet formula examples (Excel):

Truncate supplier name to 40 chars:
=LEFT(TRIM(SourceSheet!B2), 40)

Ensure payment terms maps to valid Sage value (default 30 if blank):
=IF(ISBLANK(SourceSheet!G2), 30, SourceSheet!G2)

Format date for Sage (DD/MM/YYYY):
=TEXT(SourceSheet!K2, "DD/MM/YYYY")

Strip non-numeric characters from phone:
=SUBSTITUTE(SUBSTITUTE(SUBSTITUTE(SourceSheet!H2," ",""),"-",""),"+","")

After building the staging sheet, I export it as a .csv and do one final visual check — I open it in Notepad to confirm there are no stray commas, no BOM characters at the top (these break Sage's importer), and that the header row matches the Sage template exactly.

5. Test Import in a Clean Company

Never do a first import into the live company. I always create a test company in Sage first — it takes two minutes — and run the full import there. If anything breaks, the test company is disposable.

What I check after a test import:

  1. Record count — source file had 87 suppliers, Sage now shows 87 suppliers.
  2. Spot-check 5–10 records manually against the source file (names, balances, terms).
  3. Run the Supplier Age Analysis report — opening balances should reflect correctly with the correct aging bucket based on the opening balance date.
  4. Check the Trial Balance — Creditors control account should match the sum of all supplier opening balances exactly.
The reconciliation test: Sum all supplier opening balances in Excel. Export the Sage Trial Balance after import. The Creditors Control (or Accounts Payable) account balance must equal that sum to the cent. If it doesn't, something was imported incorrectly. Do not proceed to live until this reconciles.

6. Get Sign-Off Before Going Live

Once the test import looks clean, I produce a brief validation report — usually a one-page summary — and present it to the client's accountant or financial manager. It covers:

  • Number of records imported per category
  • Total opening balances (Debtors, Creditors, Bank, Stock) with source vs. Sage comparison
  • Any records excluded and why
  • Sign-off line with date

The sign-off is not bureaucratic friction — it protects both parties. If the client later says "the opening stock value is wrong," you have a document showing they confirmed it before go-live. More importantly, it forces the right conversation to happen at the right time, before anyone has entered live transactions on top of the data.

7. Lessons From the Field

"The data you receive is never the data you were promised."

A few patterns I encountered repeatedly across 20+ implementations:

  • The duplicate supplier problem — "ABC Supplies" and "A.B.C Supplies (Pty) Ltd" are the same company. Always get a canonical name list confirmed by the client before importing.
  • The tax registration number gap — Many small businesses in Botswana don't have all supplier TRNs on record. Sage doesn't require them, but it's good practice to flag the gaps to the client rather than importing blank.
  • The mid-period cut-over — Clients sometimes want to go live mid-month. Opening balances then need to include transactions already processed in that partial period. Always cut over at month-end if you can negotiate it.
  • The stale inventory — Items that haven't moved in two years are still in the system. Ask the client to confirm the item list is current before importing; otherwise you're carrying dead stock into the new system.

Conclusion

A Sage 50c migration isn't technically complex — the tool is designed to be approachable. The complexity is in the data preparation. A systematic validation process, a clear field mapping, a reconciliation check, and a documented sign-off turn a potentially chaotic go-live into a predictable handover. The few hours you invest in validation before go-live save you days of cleanup after.
