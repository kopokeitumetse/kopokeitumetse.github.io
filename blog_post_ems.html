Building an Asset Tracker with FastAPI and SQL

2024-09-30 | 9 min read | tutorial fastapi python

Before I built the Equipment Management System, asset tracking at Leading Edge Markets was done in a shared Excel spreadsheet. It worked until it didn't — concurrent edits caused data loss, nobody had a clear picture of what was assigned to whom, and preparing for an internal audit meant a half-day of cross-referencing tabs. 150+ assets, no single source of truth.

This post walks through the architecture and key design decisions behind the EMS. I won't cover every line of code, but I'll go deep on the parts that mattered most: the data model, RBAC, the audit log, and the notification system.

1. Why FastAPI

I chose FastAPI for three reasons: it's fast to build with, the automatic OpenAPI docs are genuinely useful for a small internal tool (the operations team could explore the API themselves), and its dependency injection system makes RBAC clean to implement without a framework like Django REST Framework.

The stack was deliberately minimal:

  • FastAPI — API layer
  • SQLite (later migrated to PostgreSQL) — storage
  • SQLAlchemy — ORM and query building
  • python-jose — JWT token handling for auth
  • Jinja2 — simple HTML dashboard for non-technical staff

2. The Data Model

The core of the system is four tables. Everything else — service requests, notifications, reports — is built on top of these.

-- assets table
CREATE TABLE assets (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    asset_tag       TEXT NOT NULL UNIQUE,   -- e.g. "LEA-LT-042"
    description     TEXT NOT NULL,
    category        TEXT NOT NULL,           -- Laptop, Monitor, Printer, etc.
    status          TEXT NOT NULL DEFAULT 'Available',
    assigned_to     INTEGER REFERENCES users(id),
    assigned_date   TEXT,
    purchase_date   TEXT,
    warranty_expiry TEXT,
    notes           TEXT
);

-- users table (internal staff directory)
CREATE TABLE users (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    full_name  TEXT NOT NULL,
    email      TEXT NOT NULL UNIQUE,
    department TEXT,
    role       TEXT NOT NULL DEFAULT 'viewer'  -- viewer | technician | admin
);

-- service_requests table
CREATE TABLE service_requests (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    asset_id     INTEGER NOT NULL REFERENCES assets(id),
    requested_by INTEGER NOT NULL REFERENCES users(id),
    request_type TEXT NOT NULL,   -- Repair | Replacement | Maintenance
    description  TEXT,
    status       TEXT DEFAULT 'Open',
    created_at   TEXT DEFAULT (datetime('now')),
    resolved_at  TEXT
);

-- audit_log table — append-only, never updated or deleted
CREATE TABLE audit_log (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    actor_id    INTEGER REFERENCES users(id),
    action      TEXT NOT NULL,    -- ASSET_ASSIGNED, STATUS_CHANGED, etc.
    target_type TEXT,             -- "asset" | "service_request"
    target_id   INTEGER,
    detail      TEXT,             -- JSON blob of changed fields
    timestamp   TEXT DEFAULT (datetime('now'))
);

The audit_log is the most important table. No row is ever updated or deleted from it. Every state change — an asset being assigned, a service request being closed, a status being changed — writes a row here. This gave us full traceability for the audit and made it straightforward to produce a complete history for any asset.

3. RBAC with FastAPI Dependencies

Three roles: viewer (read-only), technician (can manage assets and service requests), admin (full access including user management). FastAPI's dependency injection makes role enforcement clean:

# auth.py
from fastapi import Depends, HTTPException, status
from fastapi.security import OAuth2PasswordBearer

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/auth/token")

def get_current_user(token: str = Depends(oauth2_scheme)):
    payload = decode_jwt(token)  # raises 401 if invalid
    return get_user_from_db(payload["sub"])

def require_role(*allowed_roles):
    def checker(user = Depends(get_current_user)):
        if user.role not in allowed_roles:
            raise HTTPException(
                status_code=status.HTTP_403_FORBIDDEN,
                detail=f"Role '{user.role}' cannot perform this action."
            )
        return user
    return checker

4. Automated Notifications

Two notification triggers were the most valuable to the operations team:

  1. Warranty expiry alerts — a daily background job checks for assets with warranty expiry within 30 days and emails the IT admin.
  2. Overdue service request alerts — any open service request older than 5 business days triggers a reminder to the assigned technician.
# scheduler.py — runs via APScheduler background task
from apscheduler.schedulers.background import BackgroundScheduler
from datetime import date, timedelta

def check_warranty_expiry():
    threshold = date.today() + timedelta(days=30)
    expiring = db.query(Asset).filter(
        Asset.warranty_expiry <= threshold,
        Asset.warranty_expiry >= date.today()
    ).all()

    if expiring:
        body = "\n".join([f"{a.asset_tag} — {a.description} (expires {a.warranty_expiry})" for a in expiring])
        send_email(
            to="it-admin@yourdomain.com",
            subject=f"[EMS] {len(expiring)} asset(s) warranty expiring within 30 days",
            body=body
        )

scheduler = BackgroundScheduler()
scheduler.add_job(check_warranty_expiry, "cron", hour=8, minute=0)
scheduler.start()
Outcome: Within the first three months of the EMS going live, the team identified 11 assets with warranties expiring in the next quarter — none of which had previously been flagged. Two were renewed before expiry. This alone justified the time spent building the system.

5. What I Would Change

  • Start with PostgreSQL, not SQLite. The migration from SQLite to PostgreSQL mid-project was straightforward but unnecessary friction. If there's any chance the data grows beyond a few hundred rows, start with Postgres.
  • Add a proper front-end framework earlier. The Jinja2 dashboard worked, but as the feature set grew, it became harder to maintain than a small React or HTMX front-end would have been.
  • Build the audit log first. I added it partway through. Retrofitting it meant going back through existing routes. If you're building anything with compliance requirements, design the audit log before you write the first route.
