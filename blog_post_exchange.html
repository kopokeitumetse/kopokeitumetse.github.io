SLA Monitoring with SQL: Building Monthly IT Reports

2025-02-07|7 min read| tutorial sql itsm

Every IT department makes promises — response within 4 hours, resolution within 8, no repeat incidents from the same root cause within 30 days. The problem isn't making those commitments; it's knowing whether you're keeping them. At Leading Edge Markets, I built a SQL-based reporting layer on top of our incident and asset tracking data that produces a monthly management report automatically. This post covers the queries and views that make it work.

1. The Data You Need First

Before you can measure SLA, you need structured incident data with at minimum: a ticket ID, a category, opened and closed timestamps, an assigned technician, and a priority level. If your system dumps this to a SQL database (or if you've built your own like the EMS), you're already in a good position.

-- Minimum schema for SLA reporting
CREATE TABLE incidents (
    id            INTEGER PRIMARY KEY,
    title         TEXT,
    category      TEXT,          -- Hardware, Software, Network, Access
    priority      TEXT,          -- P1, P2, P3, P4
    opened_at     DATETIME,
    first_response DATETIME,    -- when technician first replied
    resolved_at   DATETIME,
    technician    TEXT,
    status        TEXT           -- Open, In Progress, Resolved, Closed
);

2. Core SLA Views

Response time compliance

-- SLA targets (minutes): P1=60, P2=240, P3=480, P4=1440
CREATE VIEW v_response_compliance AS
SELECT
    id,
    priority,
    ROUND((JULIANDAY(first_response) - JULIANDAY(opened_at)) * 1440, 1) AS response_mins,
    CASE priority
        WHEN 'P1' THEN CASE WHEN response_mins <= 60   THEN 'Met' ELSE 'Breached' END
        WHEN 'P2' THEN CASE WHEN response_mins <= 240  THEN 'Met' ELSE 'Breached' END
        WHEN 'P3' THEN CASE WHEN response_mins <= 480  THEN 'Met' ELSE 'Breached' END
        WHEN 'P4' THEN CASE WHEN response_mins <= 1440 THEN 'Met' ELSE 'Breached' END
    END AS response_sla
FROM incidents
WHERE first_response IS NOT NULL;

Monthly summary report query

SELECT
    STRFTIME('%Y-%m', opened_at)                          AS month,
    COUNT(*)                                               AS total_tickets,
    SUM(CASE WHEN status = 'Resolved' THEN 1 ELSE 0 END)  AS resolved,
    ROUND(
        SUM(CASE WHEN response_sla = 'Met' THEN 1.0 ELSE 0 END) /
        COUNT(*) * 100, 1
    )                                                      AS response_sla_pct,
    category,
    technician
FROM incidents
JOIN v_response_compliance USING (id)
WHERE opened_at >= DATE('now', '-12 months')
GROUP BY month, category, technician
ORDER BY month DESC;

3. The Repeat Incident Flag

One metric I added after a pattern became obvious: incidents from the same user on the same category within 30 days. A ticket resolved and then re-opened by the same person for the same thing usually means the resolution was superficial.

SELECT
    a.id           AS original_ticket,
    b.id           AS repeat_ticket,
    a.technician,
    a.category,
    ROUND((JULIANDAY(b.opened_at) - JULIANDAY(a.resolved_at)), 1) AS days_between
FROM incidents a
JOIN incidents b
    ON  a.reported_by = b.reported_by
    AND a.category    = b.category
    AND b.opened_at   > a.resolved_at
    AND (JULIANDAY(b.opened_at) - JULIANDAY(a.resolved_at)) <= 30
ORDER BY days_between;
Real outcome: In the first quarter of using this view, we identified 3 technicians whose resolutions had high repeat rates for a specific software category. A targeted knowledge-sharing session reduced repeat incidents for that category by 60% over the following quarter.

4. Making the Report Automatic

The final step was scheduling the report export. A Python script runs on the first of each month, queries the views, formats the output as an Excel file, and emails it to management automatically — no manual intervention needed.
